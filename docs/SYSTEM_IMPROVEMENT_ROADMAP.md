# Solo Leveling Life OS — 系統改進路線圖

> 以建造者視角整理的進階優化方向：外觀與功能並重，兼顧體驗與擴展性。

---

## 一、本次已完成的調整

- **Analytics 區塊亮度**：背景改為較亮漸層（rgba(20,40,72) 等）、邊框與陰影提高對比，文字與圖例使用更亮的次要色（#8AB0CC、#B8D4F0），整體減少陰沉感。
- **Rank 階級配色**：Analytics 的 RANK 里程碑改為依設計系統 `RANK_CONFIG` 顯示各階專色：
  - **D-RANK**：系統藍 `#3F8CFF`
  - **C-RANK**：青藍 `#00C2FF`
  - **B-RANK**：紫 `#9C6BFF`
  - **A-RANK**：橙 `#FF9F43`
  - **S-RANK**：金 `#FFD700`

## 二、外觀與體驗可再強化的方向

### 2.1 動效與回饋

| 項目 | 說明 | 優先級 |
|------|------|--------|
| **微互動** | 數據卡、按鈕 hover 時輕微 scale / 光暈加強，與現有 glow-breath 一致化。 | 高 |
| **數值變動動畫** | EXP / Streak 變化時除 CountUp 外，可加短暫 pulse 或 color flash，強化「成長感」。 | 中 |
| **分頁切換** | Tasks / Summary / Analytics 切換時可做 300ms 淡入 + 輕微 translateY，減少生硬感。 | 中 |
| **雷達圖過渡** | 屬性變動時，雷達多邊形以 400–600ms 過渡到新形狀，而非瞬間跳變。 | 低 |

### 2.2 視覺層次與可讀性

| 項目 | 說明 | 優先級 |
|------|------|--------|
| **焦點區對比** | 重要區塊（如 TOP PRIORITY、RADAR）與次要區塊的背景/邊框對比可再拉開，讓「最重要的事」更突出。 | 高 |
| **字階系統** | 標題 / 副標 / 內文 / 數值建立固定字階（例如 0.5rem–1rem），全站統一，利於可讀與後續改版。 | 中 |
| **暗色模式一致性** | 確保所有新區塊（含 Analytics）在深色底下的對比度達標，必要時用 CSS 變數集中管理。 | 高 |

### 2.3 響應式與可及性

| 項目 | 說明 | 優先級 |
|------|------|--------|
| **小螢幕佈局** | Analytics 四格卡在窄屏改為 2×2 或單列；RANK 里程碑改為可橫向捲動或收合，避免擠壓。 | 高 |
| **觸控反饋** | 按鈕與可點區塊的 active 狀態（:active / touch）更明顯，方便手機操作。 | 中 |
| **無障礙** | 關鍵按鈕與狀態（如任務完成、Debuff 套用）具備適當的 aria-label / role，必要處支援鍵盤焦點。 | 中 |

## 三、功能面可延伸的方向

### 3.1 數據與目標

| 項目 | 說明 | 優先級 |
|------|------|--------|
| **週/月目標** | 使用者可設「本週目標 EXP」或「本週完成任務數」，主頁或 Analytics 顯示進度條與達成率。 | 高 |
| **歷史對比** | Analytics 的「vs last」可擴為「與上週/上月同日對比」，或簡單的「過去 7 日平均」。 | 中 |
| **匯出/備份** | 將完成紀錄、設定匯出為 JSON 或 CSV，方便備份或自建分析。 | 低 |

### 3.2 任務與習慣

| 項目 | 說明 | 優先級 |
|------|------|--------|
| **自訂任務** | 允許新增「自訂每日/挑戰任務」（名稱、預估時間、EXP），存於 localStorage 或後端。 | 高 |
| **任務提醒** | 可選的瀏覽器 Notification 或 PWA 提醒（例如每日 9:00 提醒開始第一個任務）。 | 中 |
| **重複週期** | 部分任務支援「每週 N 次」而非僅「每日一次」，系統依完成次數計算獎勵。 | 低 |

### 3.3 社交與動機

| 項目 | 說明 | 優先級 |
|------|------|--------|
| **成就分享** | 達成成就或 Rank 升級時，一鍵生成分享圖（等級、Rank、Streak、今日完成數）或短連結。 | 中 |
| **簡易統計對比** | 匿名與「同系統其他使用者」的平均 Streak / 完成率比較（若未來有後端）。 | 低 |

### 3.4 系統穩定性與架構

| 項目 | 說明 | 優先級 |
|------|------|--------|
| **狀態持久化** | 除現有 localStorage 外，可考慮關鍵狀態的防抖寫入、版本遷移腳本，避免資料損壞。 | 高 |
| **錯誤邊界** | React Error Boundary 包住主要區塊（任務列、Analytics、計時器），單一元件錯誤不導致整頁白屏。 | 高 |
| **離線提示** | 偵測 offline 時顯示小橫幅「目前為離線，資料將在連線後同步」，避免使用者誤解。 | 中 |

## 四、技術債與程式品質

- **主頁元件拆分**：`page.tsx` 體積大，可逐步拆出 `QuestTerminal`、`AnalyticsPanel`、`SummaryPanel`、`TimerOverlay` 等，利於維護與測試。
- **共用邏輯抽離**：計時、EXP 計算、Rank 判定、localStorage 讀寫等可集中成 hooks 或 small modules，減少重複與錯誤。
- **型別與常數**：將 Rank 階級、Quest 類型、設計 token 等集中到 `types.ts` / `constants.ts`，與設計系統文件對齊。

---

## 五、建議實施順序（摘要）

1. **短期（體驗與穩定性）**  
   - 維持並微調 Analytics 亮度與 Rank 配色（已完成）。  
   - 響應式：Analytics 與任務列在小螢幕的排版。  
   - 錯誤邊界與狀態持久化策略。

2. **中期（動機與目標）**  
   - 週/月目標與進度顯示。  
   - 自訂任務（至少本地）。  
   - 微互動與數值動畫統一。

3. **長期（擴展與架構）**  
   - 元件與邏輯拆分、型別與常數整理。  
   - 可選的提醒、匯出、分享等進階功能。

以上可依產品節奏與資源，擇項納入迭代計畫；外觀與 Rank 配色已依設計系統落實，其餘項目可作為後續版本的改進清單。
